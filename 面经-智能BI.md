# 面经：智能BI

## 项目简介（简历可用，1～2 句）
我做的是一个基于 AI 大模型的数据分析前端平台，用户上传 Excel 数据并输入分析目标后，系统会生成分析结论和 ECharts 可视化结果。我主要负责 React + TypeScript 前端实现，覆盖上传生成链路、历史图表管理、异步任务轮询、AI 图表安全校验和大数据量列表性能优化。

## 简历 bullet（4～6 条）
- 负责智能 BI 平台前端开发，基于 React 18、TypeScript、Umi Max 和 Ant Design Pro 搭建上传分析、历史管理与数据看板页面。
- 实现同步生成与异步任务生成两条图表生产链路，使用 `FormData` 传输文件与业务参数，并将异步任务状态串联到“我的图表”页面展示。
- 针对 AI 返回的 ECharts 配置增加 `zod` 运行时校验、危险字段清洗与错误边界兜底，降低异常图表数据导致页面渲染失败的风险。
- 开发历史图表管理与查询功能，支持搜索、分页、状态展示、图表预览和原始数据查看，提升用户对生成结果的可追溯性。
- 在原始数据弹窗中引入 `react-window` 虚拟列表，并封装智能轮询 Hook，结合指数退避和 Page Visibility API 减少无效渲染与无效请求。
- 实现可配置数据看板，支持从历史图表中选择结果、拖拽布局与本地持久化保存，增强分析结果的复用能力。

**口播版（第一人称，≥150 字，完整 STAR）**
我会讲“上传文件并生成图表”这条完整闭环，因为这是我参与度最高、也最能体现前端工程能力的一段。场景上，用户不是只点一个按钮，而是要先输入分析目标、可选图表名称和图表类型，再上传文件，最后拿到分析结论和图表结果。我的任务不是把表单拼出来就结束，而是保证提交参数正确、防止重复提交、请求结束后页面状态能稳定切换。具体动作上，我在 `AddChart` 页面里把表单和结果区拆成左右两列，提交时先清空旧结果、设置 `submitting` 锁，避免重复点击；然后把文件从 Upload 组件里取出原始 `originFileObj` 传给服务层，由服务层统一用 `FormData` 组装请求。接口成功后我再分别更新分析结论和图表 JSON，失败则只提示错误不保留脏数据。结果上，这条链路让页面交互比较清晰，也让我在面试时可以很完整地讲清楚 React 状态、异步请求、文件上传和结果渲染之间的关系。

**追问 1**：你为什么要在提交前把旧图表和旧结论清空？
**口播版（≥150 字）**：我这样做主要是为了避免“旧结果冒充新结果”的用户体验问题。场景上，图表生成是一个明显有耗时的异步操作，如果用户连续改了两次目标再提交，而页面还保留上一次的图表，就会让人误以为当前正在看的图就是这次请求的结果。我的任务其实是保证页面状态和请求状态一致。动作上，我在每次提交开始时先把 `chart` 和 `rawChartJson` 置空，再把 `submitting` 置为 `true`，让右侧区域在请求返回前只展示 loading 或空态。这样一来，即使接口慢一点，用户也能明确知道“现在没有新结果，还在生成中”。结果上，这个处理虽然代码不复杂，但很能体现我对异步 UI 一致性的关注，因为很多 Demo 只关注请求成功，不关注请求中间态和旧数据残留的问题。

**追问 2**：如果让你现在继续优化这条同步链路，你还会补什么？
**口播版（≥150 字）**：如果继续优化，我会优先补三件事。第一件是上传前校验，把文件类型、文件大小和是否选择文件的逻辑尽量前置到前端，而不是等后端报错后才反馈，这样交互会更顺。第二件是把表单类型从 `any` 收紧成更明确的请求类型，因为现在页面 `onFinish` 里参数偏宽松，虽然能跑，但类型表达不够强。第三件是增强失败态和重试策略，比如图区失败时可以给“重试生成”或“重新提交”的引导。场景上这三个优化都不属于“新功能”，但都是线上体验相关的工程细节。结果上，如果把这些点补齐，我觉得这条同步链路会更像成熟业务页面，而不是只完成 Happy Path 的课堂项目。

### Q2（频率：高）[前端/工程化] 你们为什么要同时做同步生成和异步生成两种模式？
**面试官问题**：一个图表生成接口不够吗，为什么要拆成两条链路？

**口播版（第一人称，≥150 字，完整 STAR）**
我认为拆成同步和异步两种模式，本质上是在平衡用户体验和任务耗时。场景上，小数据量分析时，用户更希望“提交完立刻看到图”；但如果数据量大或者 AI 返回慢，同步请求会让页面一直卡在等待状态，失败时也不容易给出清晰的过程反馈。我的任务是把这两类场景拆开处理，而不是强行让一个接口兼容所有体验。动作上，我保留了同步页 `AddChart`，适合快速分析；同时增加了异步页 `AddChartAsync`，用户提交后只拿到成功提示，然后跳到“我的图表”页查看任务状态。后端旁证也很清楚，异步接口会先把图表记录存成 `wait`，然后后台更新成 `running`、`succeed` 或 `failed`。结果上，这种设计让页面职责更明确，也给后面做轮询、状态展示和历史追踪提供了基础，不会把所有复杂度都压在一个阻塞式页面里。

**追问 1**：那为什么异步模式提交成功后要跳去“我的图表”，而不是停留在当前页？
**口播版（≥150 字）**：我当时这么设计，主要是因为异步模式的核心不是“当前页立即出结果”，而是“任务被成功创建且后续可追踪”。如果我让用户停留在提交页，其实页面上缺少一个天然的状态容器，因为提交页本身更适合输入表单，不适合持续展示多条任务的进度。我的任务是让用户从“提交任务”自然过渡到“查看任务”。动作上，我在提交成功后先用 message 明确告诉用户去哪里看进度，再通过 `history.push('/my_chart')` 做延迟跳转，让用户有一点视觉确认时间。结果上，这种设计把“创建任务”和“管理任务”拆成两个页面，职责更清晰，用户以后查看历史图表也只需要记住一个入口，不会在多个页面之间来回找结果。

**追问 2**：如果后面你们觉得两个入口太复杂，你会怎么收敛？
**口播版（≥150 字）**：如果要收敛，我会优先从交互层收敛，而不是马上砍掉技术实现。场景上，双入口确实可能让用户不理解什么时候该用同步、什么时候该用异步。我的思路是先保留两套后端能力，但在前端提供一个统一入口，比如用户只进一个“生成图表”页面，默认走同步；如果前端根据文件大小、预估耗时，或者后端返回“任务已转异步”，再自动切换到异步流程。这样我既保留了低耗时任务的即时反馈，又减少了用户的决策成本。结果上，这种方案对已有接口的改动更小，也更符合业务演进路径，比直接推翻重做更稳。面试时我会强调这不是现有实现，而是基于现有代码结构我会怎么继续演化。

### Q4（频率：高）[前端/安全] 你是怎么处理 AI 返回的异常或危险图表配置的？
**面试官问题**：如果 AI 返回的字符串不是合法 JSON，或者里面带了不安全内容，页面怎么办？

**口播版（第一人称，≥150 字，完整 STAR）**
这个问题我会从“先校验，再清洗，再兜底渲染”三个步骤回答。场景上，AI 输出天然不可信，最差的情况不是图表难看，而是页面直接报错或渲染出不合适的内容。我的任务是保证即使 AI 结果有问题，用户也至少能看到清晰告警，而不是整个页面白屏。动作上，第一步我做了 JSON 容错解析，解析失败时会尝试从原始字符串中截取 JSON 主体；第二步我用 zod 校验结构，如果核心字段不符合要求，就返回失败结果；第三步我还做了 `sanitize`，把可能包含函数字符串或高风险键，比如 `formatter`、`rich`、`graphic`、`bindEvent`，从对象里去掉。结果上，这样即使 AI 返回异常，`SafeChart` 也能给用户展示 warning 或 fallback 图，而不是把未知字符串直接交给图表库。

**追问 1**：为什么你还额外包了一层 `ChartErrorBoundary`？
**口播版（≥150 字）**：因为 schema 校验和运行时渲染异常不是一回事。场景上，哪怕一个对象通过了我的基础校验，也不能百分之百保证 ECharts 在渲染阶段完全不报错，比如某些边界数据组合、第三方库内部异常，仍然可能在渲染期抛出错误。我的任务是把“数据不合法”和“第三方组件渲染失败”这两类风险拆开处理。动作上，我在 `SafeChart` 里先根据 `parseChartOption()` 的结果决定是正常渲染还是降级渲染，然后无论哪种分支，最终都会包在 `ChartErrorBoundary` 里。如果图表组件内部真的抛错，就展示一个 warning 结果页，并提供重试按钮。结果上，这形成了双层保护：前一层防止坏数据进来，后一层防止第三方渲染把页面拖死。

**追问 2**：你现在这套安全处理有什么边界？
**口播版（≥150 字）**：我会比较诚实地说，这套处理主要解决的是前端消费 AI 图表 JSON 时的稳定性和基础安全性，不会把它说成完整的安全体系。场景上，当前代码重点处理的是异常 JSON、关键字段缺失、疑似函数字符串和若干危险键，但它不是一套覆盖所有 ECharts 语义层风险的白名单系统。我的任务是以较低成本拦住最常见、最容易导致页面问题的风险。动作上，我保留了较宽松的 `.passthrough()`，也说明我并没有试图穷举所有字段。结果上，这让系统更实用，但也意味着如果未来业务对安全要求更高，我会考虑做更细粒度的字段白名单、服务端校验前置，以及对 AI 输出增加更严格的格式约束。这些我会明确说成“可演进方向”，不会冒充成当前已做。

### Q5（频率：高）[前端/性能优化] 你们的轮询为什么不是固定间隔一直刷？
**面试官问题**：异步任务页为什么要单独封一个 `usePolling`，它解决了什么问题？

**口播版（第一人称，≥150 字，完整 STAR）**
我封这个 Hook 的出发点是，异步任务轮询如果写成最简单的固定 3 秒请求一次，功能上能跑，但对真实页面来说会有明显浪费。场景上，“我的图表”页里有些任务已经完成，有些任务长时间停留在 `wait` 或 `running`，而且用户还可能切到其他标签页。如果我不区分这些情况，页面就会持续打接口。我的任务是让轮询只在必要时发生。动作上，我先在页面里通过 `hasPendingCharts` 判断列表中是否存在 `wait` 或 `running` 状态，只有存在待处理任务时才启用轮询；然后在 Hook 里记录上一次状态快照，如果前后数据没变化，就通过指数退避把轮询间隔逐步放大，直到上限；一旦发现状态有变化，再把间隔重置回基础值。结果上，这样的设计能兼顾及时性和请求成本，比死板的固定轮询更像真实业务里的工程化处理。

**追问 1**：为什么你这里用 `setTimeout` 递归，而不是 `setInterval`？
**口播版（≥150 字）**：我更倾向用 `setTimeout` 递归，是因为它更容易控制异步请求完成后的下一次调度。场景上，轮询请求本身是异步的，如果用 `setInterval`，很容易出现上一轮请求还没结束，下一轮又已经触发，最终造成并发请求重叠，尤其在网络波动时问题会更明显。我的任务是确保一次轮询结束后，再根据最新状态决定下一次轮询什么时候开始。动作上，我在 `scheduleNext()` 里先清掉旧 timer，再 `setTimeout(async () => { await callback(); scheduleNext(); })`，这样请求完成后才会进入下一轮。结果上，这种写法虽然比 `setInterval` 多一点代码，但对节奏控制更稳，也更方便结合“有变化重置间隔、无变化指数退避”这类策略。

**追问 2**：Page Visibility API 在这里到底有什么价值？
**口播版（≥150 字）**：它的价值非常直接，就是减少用户看不见时的无效请求。场景上，异步任务可能要几十秒甚至更久，用户很可能提交后切到别的页面或标签，如果这时前端还在后台持续轮询，其实是在消耗浏览器和服务端资源。我的任务是让轮询跟“用户是否正在关注页面”产生联动。动作上，我在 Hook 里监听 `visibilitychange`，当 `document.visibilityState` 变成 `hidden` 时就清掉 timer；当页面重新变成 `visible` 时，再把当前间隔重置成基础值，并立即触发一次拉取，确保用户一回来看到的是最新状态。结果上，这个处理既减少了后台空转，也不会牺牲用户回到页面后的新鲜度，是一个比较划算的前端优化点。

### Q6（频率：高）[前端/性能优化] 你在项目里是怎么用 `react-window` 的？
**面试官问题**：你简历写了 `react-window`，具体用在什么地方，解决了什么问题？

**口播版（第一人称，≥150 字，完整 STAR）**
我用 `react-window` 的场景不是首页主列表，而是“我的图表”页面里查看原始数据的弹窗。场景上，图表生成成功后，用户可能想回看上传后的 CSV 数据，如果把所有行直接渲染成普通表格，数据量一大，弹窗首次打开和滚动都会变卡。我的任务是尽量在不引入复杂表格库的前提下，降低大行数渲染压力。动作上，我把弹窗组件封成 `VirtualDataTable`，先把 CSV 字符串解析成 `headers` 和 `rows`，再使用 `react-window` 的 `FixedSizeList` 渲染行数据，只让可视区域附近的行进入 DOM。标题区和表头还是正常渲染，只有数据行走虚拟列表。结果上，这样改完之后，即使行数比较多，弹窗也不会一次性挂很多 DOM 节点，比较适合“偶尔看大数据、但不需要复杂编辑能力”的场景。

**追问 1**：这个优化的边界和不足是什么？
**口播版（≥150 字）**：我会明确说它优化的是“渲染成本”，不是“全链路性能”。场景上，当前 `VirtualDataTable` 在进入虚拟列表前，还是会先把整段 CSV 用字符串切分成二维数组，这一步本身仍是一次性完成的，所以如果数据再大，解析阶段依旧可能有成本。我的任务只是把最明显的 DOM 渲染瓶颈先处理掉。动作上，我选择了固定行高的 `FixedSizeList`，因为这个场景里每行展示形式统一，实现成本低；但代价是它更适合规则列表，不适合复杂的自适应高度。结果上，我会把这个优化描述成“针对原始数据查看弹窗的轻量虚拟化”，而不会夸成“彻底解决海量数据问题”。如果面试官继续追问，我会进一步说可以往流式解析或 Worker 方向演进，但那不是当前仓库已有实现。

### Q14（频率：中高）[前端/React] 你在这个项目里是怎么理解“状态建模”的？
**面试官问题**：不是所有状态都叫状态管理，你在这个项目里怎么拆页面状态？

**口播版（第一人称，≥150 字，完整 STAR）**
我在这个项目里对状态建模的理解，是先按“生命周期”和“作用域”拆，而不是一上来就上全局状态库。场景上，上传页、历史页、看板页的状态性质其实完全不同。上传页更偏一次性表单和提交中间态；历史页则有查询参数、列表数据、总数、loading、轮询开关、数据弹窗；看板页又是本地持久化布局和已选图表集合。我的任务是让每类状态尽量待在最合适的层级。动作上，我在大多数页面里优先使用 `useState` 和 `useEffect`，把页面局部状态收在页面组件内部，比如 `submitting`、`searchParams`、`dataModalChart`；只有像当前登录用户这样的全局信息，才通过 Umi 的 `initialState` 获取。结果上，这样做让状态边界比较清晰，不会为了“看起来高级”把本地状态过度提升到全局，也方便后续维护。

**追问 1**：为什么“我的图表”页里要把 `searchParams` 独立成一个状态对象？
**口播版（≥150 字）**：因为那一页的查询行为本身就是围绕一组参数在变化，而不是某一个输入框单独变化。场景上，分页、搜索词、每页条数这些值会共同决定接口返回结果，如果拆得过散，很容易在更新时漏掉彼此关系。我的任务是让“发起一次查询”的条件在代码里足够聚合。动作上，我把 `current`、`pageSize` 和搜索名称放到同一个 `searchParams` 对象里，然后把 `loadData()` 的依赖建立在这组参数之上。这样用户搜索时重置分页、切页时保留当前搜索条件，逻辑都能围绕同一个对象完成。结果上，这是一种比较典型的后台页状态建模方式，也方便后续继续补筛选条件。

**追问 2**：那你为什么没把轮询状态也塞进全局？
**口播版（≥150 字）**：因为轮询状态强依赖当前页面的数据和可见性，它天然是页面级逻辑，不太适合升到全局。场景上，只有“我的图表”页需要根据列表里的任务状态决定是否轮询，也只有这个页面关心当前标签是否可见。我的任务是让逻辑贴近使用处，避免产生跨页面副作用。动作上，我把轮询抽成可复用 Hook，但具体启用条件 `enabled: hasPendingCharts`、状态快照 `reportDataSnapshot` 仍由页面自己控制。结果上，这样既复用了通用轮询机制，又没有把业务特定状态污染到全局层。面试时我会强调，复用和提升状态不是一回事，能抽 Hook 不代表就该做全局 store。
